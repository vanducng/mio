# P2 ‚Äî SDKs (sdk-go + sdk-py)

## Overview

Thin NATS wrappers that bake in the conventions: idempotency keys, OTel,
Prometheus metrics, schema-version checks. Convention enforcers, not
NATS replacements. After P2, gateway + AI service both consume the same
typed surface.

## Goal & Outcome

**Goal:** Publish/consume APIs typed against `mio.v1`, with idempotency, tracing, metrics built in. One Go package, one Python package, mirror surface.

**Outcome:** Gateway code (P3) imports `sdk-go` for publish; echo-consumer (P4) imports `sdk-py` for consume. No code in either calls `nats-io/nats.go` or `nats.aio` directly.

## Subject grammar (locked from research)

```
mio.<direction>.<channel_type>.<account_id>.<conversation_id>[.<message_id>]
```

Replaces the earlier `mio.<dir>.<channel>.<workspace_id>.<thread_id>` draft
in `docs/system-architecture.md` ¬ß5 ‚Äî aligns wire segments with the proto
foundation (channel_type from registry, account_id = mio UUID, conversation_id
covers DM/group/channel/thread uniformly). Architecture doc ¬ß5 to be patched
when the SDK lands.

Examples:
```
mio.inbound.zoho_cliq.<acct-uuid>.<conv-uuid>
mio.outbound.zoho_cliq.<acct-uuid>.<conv-uuid>.<msg-ulid>     # edit/delete commands
mio.outbound.slack.<acct-uuid>.<conv-uuid>
```

Use UUIDs (not slugs) so the subject stays stable across rename/display-name
changes. Subject tokens may not contain `.`; UUIDs and ULIDs are safe.

## Steps (Go)

1. `cd sdk-go && go mod init github.com/vanducng/mio/sdk-go`; add `nats.go`, `prom-client`, `otel` deps.
2. `subjects.go` ‚Äî `Inbound(channelType, accountID, conversationID)` and `Outbound(channelType, accountID, conversationID, messageID...)` returning the canonical strings. **Tested**: known input ‚Üí known output. Reject inputs with `.` or empty segments. Reject `channelType` not in `Known` (see step 7).
3. `client.go` ‚Äî `New(url, opts...)` returns a `*Client`. Options: `WithName`, `WithCreds`, `WithTracerProvider`, `WithMetricsRegistry`.
4. `publish_*.go` ‚Äî each call:
   - Build `Nats-Msg-Id`:
     - Inbound: `inb:<account_id>:<source_message_id>` ‚Äî namespace by `account_id` so two installs of the same channel can't collide.
     - Outbound: `out:<send_command.id>` ‚Äî ULID is already globally unique.
   - Inject OTel context as `traceparent` header.
   - Increment `mio_sdk_publish_total{channel_type, direction, outcome}`.
   - Call `JetStream.PublishMsg` with subject built from proto fields, not a separate arg.
   - Reject publish if `Verify(msg)` (schema_version + non-empty four-tier IDs) fails.
5. `consume_*.go` ‚Äî pull-subscribe helpers returning a `<-chan Delivery` exposing `.Ack()` / `.Nak(delay)` / `.Term()`. Durable name passed by caller; SDK does not invent durable names. Ergonomic wrapper: `c.ConsumeInbound(ctx, durable string)`. Caller owns lifecycle.
6. `version.go` ‚Äî `const SchemaVersion = 1`; `Verify(msg)` returns error on mismatch or missing `tenant_id`/`account_id`/`channel_type`/`conversation_id`.
7. `channeltypes.go` ‚Äî generated from `proto/channels.yaml` by `tools/genchanneltypes/`; `var Known = map[string]bool{...}` (only `status: active` entries). SDK rejects publish for unknown `channel_type`. Regenerated by `make proto-gen` so the SDK and adapter stay in lockstep.
8. Integration test: spin up `playground/nats/` via Make, run `go test ./sdk-go -tags=integration`.

## Steps (Python)

1. `cd sdk-py && uv init` (or poetry); deps: `nats-py`, `protobuf`, `prometheus-client`, `opentelemetry-api`, `opentelemetry-sdk`.
2. Mirror the Go API surface ‚Äî same method names, same subject builders, same idempotency semantics.
3. Tests: pytest + `pytest-asyncio`, integration-tagged.

## Success Criteria

- [ ] `make test` (Go side) and `pytest` (Python side) both pass against local NATS
- [ ] Publishing the same `(account_id, source_message_id)` twice within 2min produces one stream message (NATS dedup window working via `Nats-Msg-Id`)
- [ ] OTel trace context propagates through a publish + consume round-trip
- [ ] Both SDKs reject a message with `schema_version=2`
- [ ] Both SDKs reject a publish with empty `tenant_id` or `account_id` or `channel_type` or `conversation_id`
- [ ] Both SDKs reject a publish with `channel_type` not present (and `status: active`) in `proto/channels.yaml`
- [ ] Prometheus endpoint exposes `mio_sdk_publish_total{channel_type, direction, outcome}` and `mio_sdk_consume_total{channel_type, direction, outcome}` ‚Äî labels limited to those three; no `account_id`/`tenant_id` labels

## Risks

- **API surface drift Go ‚Üî Python** ‚Äî write the Go side first, then translate; review side-by-side before merging
- **OTel header naming** ‚Äî NATS headers must be ASCII; W3C `traceparent` is fine, but custom tags need sanitization
- **Sync vs async in Python** ‚Äî pick async-only (matches `nats-py`); document
- **Metric cardinality** ‚Äî keep label set tight: `channel_type`, `outcome`. `account_id` as label = explosion ‚Üí keep it out of labels
- **Subject token validation** ‚Äî UUIDs/ULIDs are safe but free-text would break NATS subject matching; SDK rejects any segment containing `.` or empty
