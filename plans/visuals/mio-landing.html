Why this exists

AI agents need infrastructure, not glue code.

Every team building agents rewrites the same plumbing. mio is the layer you keep when the model changes.

PROBLEM 01

Webhooks are 10 different APIs

Slack signs differently from Messenger. WhatsApp retries differently from Cliq. Google Chat throttles differently from Teams. Every platform's quirks leak into your agent code — until it's mostly glue.

PROBLEM 02

Agents crash mid-thought

Your worker OOMs while the LLM is streaming. The user sees half a reply, your tool-call result is gone, the conversation is corrupted. There's no replay.

PROBLEM 03

"Send once" is fiction

You retry on a 502. The platform queues both. The user gets two replies. Or worse: two charges, two tickets, two pings to oncall.

Capabilities

Everything an agent operator needs. Nothing more.

Twelve things mio does so your agent doesn't have to. All built on NATS JetStream — battle-tested at trillions of messages.

Exactly-once delivery

Per-message dedup keys + JetStream ack window. Your agent sees each inbound event one time, even across reconnects.

Crash-safe state

Conversation context lives in NATS KV with per-thread atomic updates. Restart the agent process; pick up the exact turn it died on.

Time-travel replay

Replay the last N days of any user/thread into a new agent. A/B a new prompt against real traffic with zero customer impact.

Streaming responses

Token-by-token to channels that support edits (Slack, Telegram, Discord); typing indicators on the rest. One agent code path.

Normalized envelope

One typed message schema across every channel. `from`, `thread`, `text`, `attachments`, `reply_to` — same shape, every adapter.

Backpressure-aware

LLM rate-limited? Channel throttling? mio holds the queue, retries with jittered backoff, and never silently drops.

Outbound idempotency

Send the same intent twice; mio sends once. Idempotency keys are first-class, not retrofitted with hashes of message bodies.

Tenant isolation

Per-customer NATS accounts and subject ACLs. Run hundreds of agents on one cluster without leaking data between tenants.

Tool-call hand-off

Agent kicks off a 60s browser/code/DB tool. mio holds the typing indicator, suspends the turn durably, resumes when done.

Cost & latency observability

Built-in OpenTelemetry spans from inbound message → agent → LLM → tools → outbound. Per-tenant token & dollar caps.

Adapter SDK

~200 lines of Go to add a new channel. Today: 10 adapters. Tomorrow: your internal CRM, voice, IVR, anything with a webhook.

Pre-send DLP hooks

Run a redactor between the agent and the wire. PII stays in your VPC; what leaves is exactly what you allow to leave.

How it fits together

One bus. Many channels. Your agents on top.

mio sits between every messaging platform and your agent runtime. Inbound events become NATS subjects; outbound replies are durable, idempotent, and traced end-to-end.

The bot that survives the demo crash

Mid-conversation, your container OOMs. The prospect is half-way through a question on WhatsApp. mio's durable thread state lets the next replica resume the exact turn — same context window, same pending tool call, same typing indicator. No "sorry, can you repeat that?"
